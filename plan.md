Phase-wise plan for your current system should be incremental, not a rewrite. You already have report extraction, SQL storage, keyword retrieval, and vector retrieval. The plan below upgrades reliability first, then RAG quality.

Phase 0: Baseline and target definition (2-3 days)
- Objective: lock a measurable baseline before changing logic.
- Work: create an evaluation set of 60-100 real doctor questions, split by metric history, latest value, trend, and mixed context.
- Work: record current outputs from SQL keyword retrieval and current vector retrieval.
- Deliverable: baseline report with accuracy, completeness, and wrong-patient leakage.
- Exit criteria: you have baseline numbers and a fixed acceptance target.

Phase 1: Data model upgrades for retrieval reliability (3-5 days)
- Objective: make SQL facts queryable without breaking current ingestion.
- Work: extend existing metric row model with normalization fields.
- Work: add fields such as canonicalKey, valueNum, valueText, unitNormalized, observedAt, parseConfidence, sourceSnippet, parserVersion.
- Work: add indexes for patient plus canonicalKey plus observedAt, and patient plus observedAt.
- Work: keep raw key, raw value, raw unit untouched for audit.
- Deliverable: migration and updated schema in schema.prisma.
- Exit criteria: no ingestion break, old rows still readable, new fields nullable and backfillable.

Phase 2: Normalization and mapping engine (5-7 days)
- Objective: convert extracted key-value rows into deterministic observations.
- Work: implement concept dictionary and alias mapping for top 20-30 metrics first.
- Work: implement parsers for numeric values, ranges, and blood pressure composite values.
- Work: define confidence rules for exact alias, fuzzy alias, and unknown key.
- Work: unresolved metrics stay stored but marked unknown, not dropped.
- Deliverable: normalization module and dictionary config, called by ingestion pipeline.
- Exit criteria: blood pressure, glucose, HbA1c, creatinine, WBC work end-to-end with high confidence.

Phase 3: Ingestion integration and backfill (4-6 days)
- Objective: run normalization automatically on every new report and old reports.
- Work: integrate normalization into report ingestion path in actions.ts.
- Work: keep existing report extraction and vector ingestion behavior.
- Work: create backfill job to normalize historical rows in batches.
- Deliverable: dual-write flow (raw plus normalized) for all future reports and a completed backfill.
- Exit criteria: at least 95 percent of historical rows mapped or flagged unknown with reason.

Phase 4: Structured retrieval v2 for doctor questions (4-6 days)
- Objective: replace keyword-only metric retrieval with intent-based structured queries.
- Work: add query planner output schema with intent, metric concepts, date window, and patient scope.
- Work: implement fixed query functions such as getMetricHistory, getLatestMetric, getMetricTrend, getAbnormalReadings.
- Work: planner resolves concept aliases to canonical keys before querying.
- Work: no free-form SQL generated by model.
- Deliverable: new structured retrieval service and server actions.
- Exit criteria: metric-history queries return complete chronologic timelines, not partial matches.

Phase 5: Patient semantic retrieval with strict filters (3-4 days)
- Objective: keep semantic context, but scoped and safe.
- Work: add patient-scoped semantic search endpoint that filters by patientId and authorization.
- Work: reuse embedding utilities in embeddings.ts, but enforce patient filters.
- Work: keep generic database search in actions.ts separate from patient search.
- Deliverable: patient semantic retrieval API returning contextual snippets.
- Exit criteria: zero cross-patient leakage in tests.

Phase 6: Hybrid fusion layer (4-6 days)
- Objective: combine deterministic metrics with narrative context for final answer quality.
- Work: route metric questions to structured-first retrieval.
- Work: call semantic retrieval in parallel only for context.
- Work: fuse with precedence rule: structured facts are primary truth, semantic is supporting evidence.
- Work: return response blocks in fixed order: metrics table, trend summary, supporting excerpts, data quality notes.
- Deliverable: hybrid retriever and answer assembler service.
- Exit criteria: blood pressure history queries consistently show full metric timeline and relevant report mentions.

Phase 7: Doctor-facing UX and response contract (3-5 days)
- Objective: make outputs clinically usable and explainable.
- Work: show timeline tables with date, value, unit, report source.
- Work: show provenance links to report and snippet.
- Work: expose confidence and unknown-metric notices.
- Deliverable: updated doctor query UI and structured API response contract.
- Exit criteria: doctor can ask one question and see deterministic values first, context second.

Phase 8: Evaluation, hardening, and rollout (4-7 days)
- Objective: production readiness.
- Work: run benchmark from Phase 0 and compare deltas.
- Work: add regression suite for top metric intents.
- Work: add observability: unknown metric rate, parse failure rate, retrieval latency, leakage alerts.
- Work: staged rollout with fallback switch to old keyword path.
- Deliverable: release report and go-live checklist.
- Exit criteria: target metrics met and safe rollback available.

Suggested acceptance targets
- Metric completeness for history queries: at least 98 percent on labeled set.
- Wrong metric value in final answer: below 1 percent.
- Cross-patient leakage: 0.
- P95 response latency: under 2.5 seconds for common queries.
- Unknown key rate trending down sprint-over-sprint.
